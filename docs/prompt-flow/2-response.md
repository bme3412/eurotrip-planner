What I'd Reorder in the Roadmap
The walkthrough's priority list is solid but I'd restructure it into sprints that tell a coherent story, because some of these items are tightly coupled:
Sprint 0: Stop the Bleeding (this week, before anything else)
Rotate all API keys. Fix the 10 contaminated cities. Delete dead files and duplicate components. Fix the footer stats and hardcoded month. These aren't features — they're hygiene. Do them before any new development.
Sprint 1: Become Discoverable (week 1-2)
Add generateMetadata() to city pages, create sitemap.xml and robots.txt, add canonical URLs, and add Open Graph tags so shared links look good. This sprint has zero user-facing UI changes but is the single biggest growth lever you have. Google takes 2-6 weeks to re-index, so the sooner this ships, the sooner you start getting organic traffic.
Sprint 2: Clean the Architecture (week 2-4)
Remove NextAuth (consolidate on Supabase Auth). Replace the 2,283-line cityData.js with build-generated JSON. Add TypeScript interfaces for your data schemas — not the whole codebase, just the data layer. This sprint makes every subsequent sprint faster.
Sprint 3: Make It Actually Work (week 4-6)
Make the suggestions API real (use visit calendar scores for the selected dates). Add persistent global navigation. Wire up the homepage date selector → real city recommendations → "plan this trip" flow. This is where the app goes from "impressive content site" to "useful tool."
Sprint 4: The Data Layer Decision (week 6-8)
This is the big one the walkthrough flagged. Migrate structured city data into Supabase tables, keeping JSON files as a seeding mechanism. Build the proper database schema (trips, saved items, user preferences). This sprint is the foundation for every lifecycle feature.
The Strategic Connections the Walkthrough Missed
The experiences scoring system is your most valuable differentiator — and it only exists for Paris. The walkthrough notes this as a content gap, but strategically, this is the feature most worth investing in. The 6-dimensional scoring rubric (Uniqueness, Cultural Significance, Experience Quality, Value for Money, Photo Appeal, Accessibility) is genuinely novel — no one else rates experiences this way. Expanding it to even your top 20 cities would give you defensible content that's hard to replicate. And it's the natural insertion point for affiliate monetization: "Book / Visit site →" on a scored experience with a 9/10 rating converts much better than a generic listing.
The Paris-only itinerary engine is both a liability and an asset. The buildParisRecommendations.js at 796 lines represents significant investment in rule-based trip planning — zone awareness, travel matrices, pace-based scheduling. That logic is genuinely sophisticated. But having it locked to Paris means your "Get My Itinerary" CTA on the homepage makes a promise only Paris can deliver. For every other city, the user hits a dead end. The generalization of this engine is where the agentic planning future lives, and it's where Bedrock/Claude comes in — instead of hand-coding travel matrices for 220 cities, you use an LLM with the city's attraction data + coordinates to generate contextual itineraries on demand.
The dual auth system is more than tech debt — it's a UX trust issue. If a user signs in with Google via NextAuth and their saved trips go to one place, then later the Supabase Auth flow kicks in and they seem logged out, you've created a confusing experience that makes people not trust the "save" feature. This directly impacts the lifecycle features — every one of them (drip emails, trip briefings, notifications) depends on a user having a single, reliable account.
The cityData.js problem is a performance issue and a content accuracy issue. That 2,283-line file is manually maintained. That means when you regenerate data for a city, you have to remember to update the hardcoded array too. It will drift. It probably already has. The fix (build-generated JSON from manifest + overview files) also gives you a single source of truth, which matters enormously when you start personalizing recommendations.
New Ideas From Reading the Walkthrough
A few things I didn't suggest before that the codebase analysis makes clear:
1. A data validation CI step. You should have a script that runs on every PR/push that validates: every city directory has correctly-named files, every overview has required fields, every attraction has coordinates, no tirana_* files exist outside the Tirana directory. This prevents the contamination problem from ever recurring. It's also good Bedrock/AI engineering practice — when you're generating content at scale with AI, you need automated quality gates.
2. The preview themes are a hidden gem. You have 5 design themes (aurora, editorial, glass, metro, noir) sitting in /preview/. These could become a user customization feature — "choose your guide style" — or more practically, they're useful for A/B testing which visual treatment converts better. Don't delete them; move them into a feature flag system.
3. The monthly data architecture is actually well-designed. The half-month granularity with weather, events, tourism levels, and practical tips is more detailed than anything competitors offer. The lazy loading pattern (SSR current + next month, client-fetch the rest) is smart. This is one area where the engineering matches the ambition.
4. Consider a "city data completeness" dashboard. You have 220 cities at wildly different quality levels. A simple internal page that shows, for each city: does it have correct file naming? ✓/✗. Does it have experiences scoring? ✓/✗. Does it have coordinates for all attractions? ✓/✗. Monthly data complete? ✓/✗. This lets you prioritize content generation and track progress toward consistent quality.
